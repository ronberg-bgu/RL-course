# Assignment 1 — Deterministic Box Pushing Pipeline

This repository contains an automated pipeline that uses an LLM (Gemini) to generate classical planning logic (PDDL) for a multi-agent Box Pushing grid world, translates that logic into a visual format, and solves it using the Fast Downward engine.

## 📁 File Overview

* **`exercises/ex1/main.py`**: The master orchestration script. It runs the entire pipeline sequentially (LLM Generation ➔ ASCII Parsing ➔ Solving & Visualization). It also uses a custom `Tee` class to automatically log all terminal output.
* **`exercises/ex1/llm_pipeline.py`**: Handles the connection to the Gemini API. This file contains the master prompt that dictates the strict physical rules of the environment and extracts the generated PDDL blocks into local files.
* **`exercises/ex1/pddl_to_map.py`**: A custom Regex parsing engine. It reads the generated `problem.pddl`, extracts the coordinates of the walls, agents, regular boxes, heavy boxes, and goals, and translates them back into a 7x7 ASCII grid for the Pygame visualizer.
* **`exercises/ex1/pddl/domain.pddl` & `exercises/ex1/pddl/problem.pddl`**: The final, mathematically sound classical planning files generated by the LLM. 
* **`exercises/ex1/planner_output.txt`**: The automated log file capturing the Fast Downward engine trace, the generated 42-step plan, and the execution trace from the visualizer.

## 🚀 How to Run

1. **Install Dependencies:** Ensure your virtual environment is active, then install the required packages:
   ```bash
   pip install -r requirements.txt
   ```

2. **API Key Setup:** Ensure you have a `.env` file in the root directory containing your Gemini API key:
   ```text
   GEMINI_API_KEY=your_api_key_here
   ```

3. **Execute the Pipeline:** From the root directory of the repository (`RL-course`), run the master script:
   ```bash
   python .\exercises\ex1\main.py
   ```
   *(Note for Mac/Linux users: Use `python exercises/ex1/main.py`)*

   **Pro-Tip for Demoing:** If you already have valid PDDL files saved in `exercises/ex1/pddl/` and want to test without consuming your API quota, you can comment out "Phase 1" inside `main.py` to seamlessly use the cached files!

## 🧠 Prompting Strategy & LLM Constraints

To bridge the gap between mathematical PDDL logic and the strict physics of the Python visualizer, we engineered our prompt with highly specific constraints:

**1. Strict Typing over Union Types:**
Standard PDDL allows for union types (e.g., `?obj - (or agent box)`), but the `unified_planning` Python parser often rejects these. We forced the LLM to abandon generic parameters and explicitly define strictly-typed positional predicates (`agent-at`, `box-at`, `heavybox-at`).

**2. Parameter Order Locking:**
Because the legacy Pygame visualizer extracts target coordinates by their index in the action string, we explicitly defined the exact parameter order the LLM was allowed to use. For example, we mandated that the target coordinate (`?to`) must always be the 3rd parameter in actions like `push-small` and `move`, preventing the visualizer from extracting the wrong tile.

**3. The "Straight-Line" Physics Fix:**
The most critical choice in our prompt was overcoming the "Magical Pull" bug. Under standard `adjacent` rules, an LLM might mathematically command an agent to push a box into the cell the agent is currently standing on (effectively pulling it backwards). 
To enforce video-game physics, we completely removed `adjacent` from the push preconditions and forced the LLM to use a custom `(straight ?loc1 ?loc2 ?loc3)` predicate. We required the LLM to pre-map all valid 3-cell lines in the initial state, physically restricting the agents to only push boxes in a forward, straight-line trajectory.

**4. Hard Grid Boundaries:**
To prevent the LLM from hallucinating tiles off the edge of the board during its straight-line calculations (e.g., `loc_6_1` or `loc_1_0`), we implemented a strict boundary rule forbidding the generation of any coordinates outside the `1-5` range.
